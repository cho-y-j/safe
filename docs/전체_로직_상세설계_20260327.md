# SafePulse 전체 로직 상세 설계

## 시스템 개요

공항 작업자가 Galaxy Watch를 착용하고, 개인 바이탈(심박/SpO₂/체온)을 실시간 모니터링.
이상 감지 시 본인에게 먼저 알리고, 무응답이면 주변 동료에게 P2P BLE로 거리비례 경보.
관제 대시보드에서 전체 현황 모니터링.

```
[사고자 워치] ──BLE P2P──→ [주변 동료 워치] (서버 없이 작동)
      │                         │
      │←──Wearable BT──→ [사고자 폰]    [동료 폰] ←──Wearable BT──→│
      │                         │
      └────────WiFi────────→ [서버] ←────Socket.io────→ [대시보드]
```

### 3자 관점 (절대 혼동 금지)
- **발신자(사고자)**: 이상이 감지된 본인. 최대 소리+진동으로 위치를 알림
- **수신자(주변 동료)**: BLE로 사고를 수신한 사람. 거리별 진동으로 사고자를 찾아감
- **관제자**: 대시보드에서 전체 상황 모니터링. 지도에서 사고자 위치 확인

## 배터리 최적화 전략 (핵심)

### 설계 철학
**배터리 소모 최소화하면서 이상 신호는 빠르게 잡는 것이 핵심.**
센서는 빠르게 체크하되, 네트워크 전송만 평상시 느리게 → 이벤트 시 즉시.

### 배터리 소모 비중
| 항목 | 소모 | 비중 | 설명 |
|------|------|------|------|
| HR 센서 읽기 | 매우 적음 | 5% | OS가 저전력으로 관리, 빠르게 읽어도 부담 없음 |
| 가속도계 | 적음 | 10% | 항상 켜짐 (낙상 즉시 감지 필수) |
| **서버 HTTP 전송** | **매우 큼** | **50%** | **WiFi/LTE 통신이 배터리 최대 주범** |
| 폰 Wearable 전송 | 적음 | 10% | BLE 기반이라 부담 적음 |
| BLE 광고/스캔 | 적음 | 15% | P2P 수신 필수 |
| 화면/진동/소리 | 이벤트 시만 | 10% | 평상시 꺼짐 |

**결론: 서버 전송 빈도만 줄이면 센서는 빠르게 체크해도 배터리 충분.**

### 최적화 구조
```
┌─────────────────────────────────────────────────────┐
│                    평상시                             │
│  센서 체크: 15초 (빠르게) → 배터리 거의 안 먹음        │
│  서버 전송: 5분에 1회 → 배터리 절약 핵심              │
│  폰 전송: 30초 (BLE라 부담 적음)                     │
│  BLE 광고: LOW_POWER                                │
│  예상 배터리: 24시간+ (8시간 근무 충분)               │
├─────────────────────────────────────────────────────┤
│                  이벤트 발생 시                        │
│  센서 체크: 1초 (최대 속도)                           │
│  서버 전송: 즉시 (1초마다)                            │
│  폰 전송: 즉시 (1초마다)                             │
│  BLE 광고: HIGH (최대출력) + LOW_LATENCY              │
│  진동+소리: 최대 (발신자는 위치 찾기용)                │
│  예상 배터리: 4시간 (긴급 상황이니 충분)               │
└─────────────────────────────────────────────────────┘
```

**정상인 데이터를 서버에 자주 보낼 이유 없음 — 이벤트 시에만 즉시 전송.**
**폰 전송은 BLE 기반이라 30초마다 보내도 배터리 부담 없음.**
**센서는 빠르게 체크해야 이상을 빨리 잡음 — 전송만 느리게 하면 배터리 절약.**

## 센서 동작 방식

### 센서별 동작
| 센서 | 동작 | 간격 | 비고 |
|------|------|------|------|
| 심박(HR) | OS가 관리, 항상 실시간 | ~3초 (SensorEventListener) | 모니터링 간격과 무관 |
| 가속도계 | 항상 실시간 (낙상 감지용) | ~60ms (UI 모드) | 모니터링 간격과 무관 |
| SpO₂ | Health Services API | 요청 시 | Watch 5에서 지원 |

**핵심: 심박과 가속도계는 모니터링 간격과 무관하게 항상 실시간 동작.**
**낙상은 가속도계로 즉시 감지 — 모니터링 루프를 기다리지 않음.**
**모니터링 루프(evaluateState)는 센서값을 판단하는 주기일 뿐, 센서 자체는 항상 켜져 있음.**

### 모니터링 주기 (evaluateState)
```
평상시: 5초마다 evaluateState() 실행
  → 센서값 판단 (HR/SpO₂/체온 이상 여부)
  → 배터리 소모 거의 없음 (센서는 이미 읽고 있고, 판단 로직만 실행)

이벤트 시: 1초마다 evaluateState() 실행
  → EMERGENCY 상태에서 서버/폰에 즉시 전송

※ 센서 자체는 항상 실시간 (HR 3초, 가속도 60ms)
※ evaluateState()는 그 값을 "판단"하는 것뿐 → 5초면 충분히 빠름
※ 낙상은 가속도계 이벤트로 즉시 감지 → evaluateState 주기와 무관
```

**핵심: 워치가 빠르게 체크하는 건 배터리 안 먹음. 서버 전송만 줄이면 됨.**
**5초마다 체크 → 이상 발견 즉시 신호 전송 → 수신자 실시간 거리 추적**

### 전송 주기 정리
| 항목 | 평상시 | 이벤트 시 | 이유 |
|------|--------|----------|------|
| 서버 HTTP 전송 | **5분에 1회** | **즉시 (1초)** | 배터리 50% 차지 — 평상시 최소화 |
| 폰 Wearable 전송 | 30초 | 즉시 (1초) | BLE라 부담 없지만 불필요한 전송 줄임 |
| BLE 광고 | LOW_POWER | HIGH (최대출력) | 평상시 존재 알림만, 이벤트 시 최대 |

## 개인화 학습 (TinyML 시뮬레이션)

### 베이스라인 구조
```
안정 시: restHrMean ± restHrStd (가만히 있을 때)
활동 시: activeHrMean ± activeHrStd (움직일 때)
SpO₂: spo2Mean ± spo2Std
체온: baselineTempMean ± baselineTempStd
```

### 학습 방법
- EMA (지수이동평균, α=0.05)로 서서히 업데이트
- 안정/활동 분리: 가속도계 활동량 > 1.5 → 활동, 이하 → 안정
- 최소 24샘플 모이면 베이스라인 준비 완료

### "괜찮아요" 피드백 학습
```
사용자가 "괜찮아요" 누름
→ "현재 내 심박/SpO₂/체온은 나에게 정상이다"
→ 현재값을 5x 가중으로 히스토리에 추가
→ EMA 업데이트 → 경보 범위가 해당 방향으로 확장
→ 다음에 같은 수치에서 경보 안 남 (개인화됨)
```

### "오작동 종료" — 학습 안 함
```
사용자가 "오작동 종료" 누름
→ 센서 오류/일시적이라 판단
→ 학습에 반영하지 않음
→ 경보 범위 변화 없음
→ 다음에 같은 수치에서 다시 경보 남 (둔감해지지 않음)
```

### 직업별 경보 범위 (연구 기반: NIOSH HRR%, PMC 실측)
| 직업 | HR 상한 | HR 하한 | SpO₂ 범위 | 체온 범위 |
|------|---------|---------|----------|----------|
| 사무/관제 | 안정+40 | 안정-30 | -3% | ±0.5°C |
| 경량 작업 | 안정+55 | 안정-30 | -4% | ±0.8°C |
| 중량 작업 | 안정+65 | 안정-30 | -5% | ±1.0°C |
| 야외 고강도 | 안정+80 | 안정-30 | -6% | ±1.5°C |

### 저장/복원
```
SharedPreferences에 전체 베이스라인 저장:
  baselineHR, restHrStd, activeHrMean, activeHrStd
  spo2Mean, spo2Std, baselineTempMean, baselineTempStd
앱 재시작 시 복원 → 학습 연속성 유지
서버에도 30분마다 동기화 → 워치 교체 시 서버에서 복원
```

## 이상 감지 흐름

### NORMAL → MILD_ANOMALY
```
다음 중 하나라도 해당되면 이상 감지:
  ① HR > 개인 안정평균 + 직업별 범위 (예: 72 + 55 = 127)
  ② HR < 개인 안정평균 - 30 (예: 72 - 30 = 42)
  ③ HR >= 180 (절대 상한, 직업 무관)
  ④ HR < 40 (절대 하한, 서맥)
  ⑤ SpO₂ < 개인평균 - 직업별 범위 (예: 98 - 4 = 94%)
  ⑥ 체온 > 개인평균 + 직업별 범위 (예: 36.5 + 0.8 = 37.3°C)
  ⑦ 심박 급락 (이전 대비 20bpm 이상 하락)

학습 초기 (+20% 여유): 샘플 60개 미만이면 상한에 20% 마진 추가
```

### 낙상 감지 (가속도계 — 실시간, 모니터링 간격 무관)
```
1단계: 자유낙하 감지 — 가속도 < 5.0 (정상 9.81)
2단계: 2초 이내 충격 감지 — 가속도 > 15.0
조건: HR=0 연속 5회 이상이면 워치 벗는 중 → 낙상 무시
결과: FALL_DETECTED → 5초 확인 대기 → 무응답 시 EMERGENCY

※ 임계값은 테스트용으로 완화됨 (실전: FREE_FALL=3.0, IMPACT=25.0)
```

### 잠 vs 응급 구분
```
30초 이상 무움직임 + 바이탈 정상 → 수면 의심 (SLEEP_SUSPECTED)
30초 이상 무움직임 + 바이탈 이상 → 응급 (EMERGENCY)
수면 의심 시: 부드러운 반복 알림 (강한 경보 아님)
```

## 이벤트 발생 시 흐름 (시간순)

### 1. 이상 감지 (0초)
```
evaluateState() → NORMAL → MILD_ANOMALY
→ 즉시 WAITING_ACK으로 전환
→ notifyWorker() 호출:
  - 진동 1회 (500ms)
  - fullScreenIntent → AckAlertActivity 실행 (화면 자동 켜짐)
  - 비프 3회
  - 모니터링 1초 간격으로 전환
```

### 2. AckAlertActivity (발신자 화면)
```
┌─────────────────┐
│  괜찮으세요?     │
│                 │
│ [✅ 괜찮아요]   │  학습 O + 경보 해제
│ [⏹ 오작동 종료] │  학습 X + 경보 해제
│ [🚨 즉시 호출]  │  즉시 P2P+서버 경보
│                 │
│ 무응답 시 자동경보│
└─────────────────┘
```

### 3-A. "괜찮아요" 누른 경우
```
→ state = ACKNOWLEDGED
→ ackCooldownUntil = now + 30초
→ 현재 HR/SpO₂/체온 5x 가중 학습 (이 값은 나에게 정상)
→ emergencyVibrator.cancel() (진동 즉시 정지)
→ stopEmergencyBeep() (비프 즉시 정지)
→ BLE cancelEmergency() → "N" 광고 (수신자 자동 정지)
→ 30초간 재에스컬레이션 완전 차단
→ 30초 후 HR 정상이면 NORMAL, 절대상한이면 재알림
```

### 3-B. "오작동 종료" 누른 경우
```
→ state = NORMAL (바로 정상)
→ ackCooldownUntil = now + 30초
→ 학습 안 함 (경보 범위 변화 없음)
→ 진동/비프 즉시 정지
→ BLE cancelEmergency()
```

### 3-C. "즉시 호출" 누른 경우
```
→ state = EMERGENCY
→ BLE broadcastEmergency() → "SPW-001E" 최대출력
→ emergencyVibrate() → 최대 볼륨 연속 비프+진동 (위치 찾기용)
→ 폰+서버에 즉시 전송
```

### 3-D. 무응답 (테스트 5초, 실전 300초)
```
→ isSleeping() 체크
  - 바이탈 정상 + 무움직임 30초 → SLEEP_SUSPECTED (부드러운 알림)
  - 그 외 → EMERGENCY
```

## EMERGENCY 상태 (사고 확정)

### 발신자(사고자) 워치
```
최초 1회 (isEmergencyAlarmActive 플래그):
  1. AckAlertActivity 표시 (fullScreenIntent, CLEAR_TOP)
  2. BLE "SPW-001E" 최대출력 광고 시작
  3. 최대 볼륨 연속 비프 (500ms 비프 + 300ms 무음 반복, 무한)
  4. 최대 강도 연속 진동 (무한 반복)
  ※ "괜찮아요"/"오작동 종료" 누를 때까지 절대 안 멈춤
  ※ 소리로 사고자 위치를 찾을 수 있게 최대 볼륨

매 1초 (상태 업데이트):
  1. 폰에 Wearable 전송 (실시간 바이탈)
  2. 서버에 HTTP 전송 (status="danger")
  3. 대시보드 자동 반영

90초 후:
  → 119 자동 연동 (TODO)
```

### 발신자(사고자) 폰 (S22)
```
Wearable Data Layer로 수신:
  - 긴급 카드 표시 (HR + SpO₂ + 상태 + 경과시간)
  - [✅ 문제 없음] → 워치에 phone_ack → onAcknowledge()
  - [⏹ 오작동 멈추기] → 워치에 phone_dismiss → onDismiss()
  - [🚨 119 신고] → 전화 다이얼
  ※ 지도 버튼 없음 (사고자 본인에게 지도는 무의미)
```

## P2P BLE 수신 (주변 동료)

### 수신자 워치 (Watch 5)
```
BLE 스캔에서 "SPW-001E" 수신:
  1. RSSI → 거리 계산 (경로손실모델: TX_POWER=-65, N=2.5)
  2. 5구간 zone 분류 (ZONE1~ZONE5)
  3. 거리 브로드캐스트 전송 → P2pAlertActivity 실시간 갱신
  4. zone 변경 시 → 진동 패턴 즉시 교체
  5. 첫 1회: fullScreenIntent → P2pAlertActivity 실행 (화면 자동 켜짐)
```

### 5구간 거리별 진동+비프 (수신자)
| 구간 | RSSI | 거리 | 진동 간격 | 비프 간격 | 색상 |
|------|------|------|----------|----------|------|
| ZONE1 | >-50 | ~1m | 100ms on, 50ms off | 200ms | 빨강 |
| ZONE2 | -50~-60 | ~3m | 200ms on, 300ms off | 500ms | 주황 |
| ZONE3 | -60~-67 | ~5m | 200ms on, 1초 off | 1초 | 노랑 |
| ZONE4 | -67~-75 | ~10m | 200ms on, 2초 off | 2초 | 파랑 |
| ZONE5 | <-75 | 10m+ | 200ms on, 4초 off | 4초 | 회색 |

**핵심: 진동 세기는 모두 MAX. 간격만 다르게 → 체감 확실**
**가까이 갈수록 빠르게 → 소리+진동 방향으로 사고자를 찾아감**
**거리는 매 BLE 수신마다 실시간 갱신 (30초 제한 없음)**

### P2pAlertActivity (수신자 화면)
```
┌─────────────────┐
│  🚨 긴급!       │
│  조영진          │  ← 사고자 이름 (서버에서 자동 로드)
│  ~5m            │  ← 실시간 거리 (매 BLE 수신마다 갱신)
│                 │
│ [🏃 응답함]     │  → 진동+비프 정지 + Activity 종료
│ [도움불가]       │  → 진동+비프 정지 + Activity 종료
└─────────────────┘
```

### 거리 실시간 갱신 구조
```
BLE 수신 → 거리 계산 → 매번:
  ① P2P_DISTANCE 브로드캐스트 → P2pAlertActivity가 수신 → 화면 거리 업데이트
  ② zone 비교 → zone 변경 시 → 진동 패턴 즉시 교체 + 비프 간격 변경
  ③ zone 동일 시 → 화면만 업데이트 (진동 패턴 유지)
```

### 경보 해제 조건
```
① 수신자가 "응답함"/"도움불가" 누름 → suppressedWorkerIds에 추가 (30초간 재알림 방지)
② 발신자가 "괜찮아요"/"오작동 종료" 누름 → BLE "N" 광고 → 수신자 자동 정지
③ 강제 종료
```

## 수신자 폰 (Note9)

### BridgeService (BLE 스캔)
```
BLE에서 "SPW-001E" 수신:
  1. isMyWatch 체크 (내 워치 ID와 비교)
     - 내 워치면 → 발신자 폰 역할 (긴급 카드)
     - 타 워치면 → 수신자 폰 역할 (아래 처리)
  2. 진동 + 비프 (10초 중복 방지)
  3. MainActivity 실행 (p2p_emergency intent)
  4. P2P_EMERGENCY 브로드캐스트 → p2pCard 표시
  5. 서버에서 사고자 위치 조회 → EmergencyMapActivity (지도) 실행
```

### 수신자 폰 화면
```
┌─────────────────────────┐
│ 🚨 주변 긴급! W-001     │
│                         │
│ [📍 사고자 위치 보기]    │ → 지도 (사고자 위치 빨간 마커)
│ [✅ 확인 (알림 종료)]    │ → 진동 정지 + 카드 닫기
└─────────────────────────┘
```

**지도는 수신자(도와줄 사람)에게만 — 사고자 본인에게 지도는 무의미**

## 통신 경로 (이중화)

### 경로 1: P2P BLE (서버 없이 작동 — 핵심)
```
사고자 워치 → BLE 광고 "SPW-001E" → 주변 워치 수신
  - 서버/WiFi 없어도 작동
  - 30m 반경 (실내 20-40m)
  - 거리비례 진동 (특허 핵심)
```

### 경로 2: 워치 → 서버 (WiFi)
```
사고자 워치 → POST /api/workers/:id/sensor (status="danger")
  → realWatchData 업데이트 → workers:update → 대시보드
사고자 워치 → POST /api/alerts/emergency
  → Socket.io "emergency" → 대시보드 즉시 팝업
```

### 경로 3: 워치 → 폰 → 서버 (WiFi 없을 때)
```
사고자 워치 → Wearable Data Layer → 폰
  → 폰이 서버에 중계 (WearableListenerService)
  → 폰 BridgeService도 BLE 스캔으로 타 워치 감지
```

## HR=0 처리 (워치 벗음/센서 접촉 불량)

```
HR=0 연속 3회:
  - 이상 상태(MILD_ANOMALY/WAITING_ACK/FALL_DETECTED)에서 → WAITING_ACK (위험 가능)
  - 정상 상태에서 → WATCH_REMOVED (조용히)

HR=0 연속 10회 (30초+):
  → 절전 모드 (모니터링 60초 간격)

다시 HR > 0 감지:
  → NORMAL 복귀 + 모니터링 정상화

낙상 감지 + HR=0:
  - HR=0 연속 5회 이상 → 워치 벗는 중 → 낙상 무시
  - HR=0 연속 4회 이하 → 일시적 → 낙상 판정 유지
```
