# SafePulse 전체 로직 상세 설계

## 시스템 개요

공항 작업자가 Galaxy Watch를 착용하고, 개인 바이탈(심박/SpO₂/체온)을 실시간 모니터링.
이상 감지 시 본인에게 먼저 알리고, 무응답이면 주변 동료에게 P2P BLE로 거리비례 경보.
관제 대시보드에서 전체 현황 모니터링.

```
[사고자 워치] ──BLE P2P──→ [주변 동료 워치] (서버 없이 작동)
      │                         │
      │←──Wearable BT──→ [사고자 폰]    [동료 폰] ←──Wearable BT──→│
      │                         │
      └────────WiFi────────→ [서버] ←────Socket.io────→ [대시보드]
```

### 3자 관점 (절대 혼동 금지)
- **발신자(사고자)**: 이상이 감지된 본인. 최대 소리+진동으로 위치를 알림
- **수신자(주변 동료)**: BLE로 사고를 수신한 사람. 거리별 진동으로 사고자를 찾아감
- **관제자**: 대시보드에서 전체 상황 모니터링. 지도에서 사고자 위치 확인

## 배터리 최적화 전략 (핵심)

### 설계 철학
**배터리 소모 최소화하면서 이상 신호는 빠르게 잡는 것이 핵심.**
센서는 빠르게 체크하되, 네트워크 전송만 평상시 느리게 → 이벤트 시 즉시.

### 배터리 소모 비중
| 항목 | 소모 | 비중 | 설명 |
|------|------|------|------|
| HR 센서 읽기 | 매우 적음 | 5% | OS가 저전력으로 관리, 빠르게 읽어도 부담 없음 |
| 가속도계 | 적음 | 10% | 항상 켜짐 (낙상 즉시 감지 필수) |
| **서버 HTTP 전송** | **매우 큼** | **50%** | **WiFi/LTE 통신이 배터리 최대 주범** |
| 폰 Wearable 전송 | 적음 | 10% | BLE 기반이라 부담 적음 |
| BLE 광고/스캔 | 적음 | 15% | P2P 수신 필수 |
| 화면/진동/소리 | 이벤트 시만 | 10% | 평상시 꺼짐 |

**결론: 서버 전송 빈도만 줄이면 센서는 빠르게 체크해도 배터리 충분.**

### 최적화 구조
```
┌─────────────────────────────────────────────────────┐
│                    평상시                             │
│  센서 체크: 15초 (빠르게) → 배터리 거의 안 먹음        │
│  서버 전송: 5분에 1회 → 배터리 절약 핵심              │
│  폰 전송: 30초 (BLE라 부담 적음)                     │
│  BLE 광고: LOW_POWER                                │
│  예상 배터리: 24시간+ (8시간 근무 충분)               │
├─────────────────────────────────────────────────────┤
│                  이벤트 발생 시                        │
│  센서 체크: 1초 (최대 속도)                           │
│  서버 전송: 즉시 (1초마다)                            │
│  폰 전송: 즉시 (1초마다)                             │
│  BLE 광고: HIGH (최대출력) + LOW_LATENCY              │
│  진동+소리: 최대 (발신자는 위치 찾기용)                │
│  예상 배터리: 4시간 (긴급 상황이니 충분)               │
└─────────────────────────────────────────────────────┘
```

**정상인 데이터를 서버에 자주 보낼 이유 없음 — 이벤트 시에만 즉시 전송.**
**폰 전송은 BLE 기반이라 30초마다 보내도 배터리 부담 없음.**
**센서는 빠르게 체크해야 이상을 빨리 잡음 — 전송만 느리게 하면 배터리 절약.**

## 센서 동작 방식

### 센서별 동작
| 센서 | 동작 | 간격 | 비고 |
|------|------|------|------|
| 심박(HR) | OS가 관리, 항상 실시간 | ~3초 (SensorEventListener) | 모니터링 간격과 무관 |
| 가속도계 | 항상 실시간 (낙상 감지용) | ~60ms (UI 모드) | 모니터링 간격과 무관 |
| SpO₂ | Health Services API | 요청 시 | Watch 5에서 지원 |

**핵심: 심박과 가속도계는 모니터링 간격과 무관하게 항상 실시간 동작.**
**낙상은 가속도계로 즉시 감지 — 모니터링 루프를 기다리지 않음.**
**모니터링 루프(evaluateState)는 센서값을 판단하는 주기일 뿐, 센서 자체는 항상 켜져 있음.**

### 모니터링 주기 (evaluateState)
```
평상시: 5초마다 evaluateState() 실행
  → 센서값 판단 (HR/SpO₂/체온 이상 여부)
  → 배터리 소모 거의 없음 (센서는 이미 읽고 있고, 판단 로직만 실행)

이벤트 시: 1초마다 evaluateState() 실행
  → EMERGENCY 상태에서 서버/폰에 즉시 전송

※ 센서 자체는 항상 실시간 (HR 3초, 가속도 60ms)
※ evaluateState()는 그 값을 "판단"하는 것뿐 → 5초면 충분히 빠름
※ 낙상은 가속도계 이벤트로 즉시 감지 → evaluateState 주기와 무관
```

**핵심: 워치가 빠르게 체크하는 건 배터리 안 먹음. 서버 전송만 줄이면 됨.**
**5초마다 체크 → 이상 발견 즉시 신호 전송 → 수신자 실시간 거리 추적**

### 전송 주기 정리
| 항목 | 평상시 | 이벤트 시 | 이유 |
|------|--------|----------|------|
| 서버 HTTP 전송 | **5분에 1회** | **즉시 (1초)** | 배터리 50% 차지 — 평상시 최소화 |
| 폰 Wearable 전송 | 30초 | 즉시 (1초) | BLE라 부담 없지만 불필요한 전송 줄임 |
| BLE 광고 | LOW_POWER | HIGH (최대출력) | 평상시 존재 알림만, 이벤트 시 최대 |

## 개인화 학습 (TinyML 시뮬레이션)

### 베이스라인 구조
```
안정 시: restHrMean ± restHrStd (가만히 있을 때)
활동 시: activeHrMean ± activeHrStd (움직일 때)
SpO₂: spo2Mean ± spo2Std
체온: baselineTempMean ± baselineTempStd
```

### 학습 방법
- EMA (지수이동평균, α=0.05)로 서서히 업데이트
- 안정/활동 분리: 가속도계 활동량 > 1.5 → 활동, 이하 → 안정
- 최소 24샘플 모이면 베이스라인 준비 완료

### "괜찮아요" 피드백 학습
```
사용자가 "괜찮아요" 누름
→ "현재 내 심박/SpO₂/체온은 나에게 정상이다"
→ 현재값을 5x 가중으로 히스토리에 추가
→ EMA 업데이트 → 경보 범위가 해당 방향으로 확장
→ 다음에 같은 수치에서 경보 안 남 (개인화됨)
```

### "오작동 종료" — 학습 안 함
```
사용자가 "오작동 종료" 누름
→ 센서 오류/일시적이라 판단
→ 학습에 반영하지 않음
→ 경보 범위 변화 없음
→ 다음에 같은 수치에서 다시 경보 남 (둔감해지지 않음)
```

### 직업별 경보 범위 (연구 기반: NIOSH HRR%, PMC 실측)
| 직업 | HR 상한 | HR 하한 | SpO₂ 범위 | 체온 범위 |
|------|---------|---------|----------|----------|
| 사무/관제 | 안정+40 | 안정-30 | -3% | ±0.5°C |
| 경량 작업 | 안정+55 | 안정-30 | -4% | ±0.8°C |
| 중량 작업 | 안정+65 | 안정-30 | -5% | ±1.0°C |
| 야외 고강도 | 안정+80 | 안정-30 | -6% | ±1.5°C |

### 저장/복원
```
SharedPreferences에 전체 베이스라인 저장:
  HR: baselineHR, restHrStd, activeHrMean, activeHrStd
  SpO₂: spo2Mean, spo2Std
  체온: baselineTempMean, baselineTempStd
  가속도: accelMean, accelStd  ← 2026-03-28 추가 (낙상 감지 적응형)
앱 재시작 시 복원 → 학습 연속성 유지
서버에도 30분마다 동기화 → 워치 교체 시 서버에서 복원
```

## 이상 감지 흐름

### NORMAL → MILD_ANOMALY
```
다음 중 하나라도 해당되면 이상 감지:
  ① HR > 개인 안정평균 + 직업별 범위 (예: 72 + 55 = 127)
  ② HR < 개인 안정평균 - 30 (예: 72 - 30 = 42)
  ③ HR >= 180 (절대 상한, 직업 무관)
  ④ HR < 40 (절대 하한, 서맥)
  ⑤ SpO₂ < 개인평균 - 직업별 범위 (예: 98 - 4 = 94%)
  ⑥ 체온 > 개인평균 + 직업별 범위 (예: 36.5 + 0.8 = 37.3°C)
  ⑦ 심박 급락 (이전 대비 20bpm 이상 하락)

학습 초기 (+20% 여유): 샘플 60개 미만이면 상한에 20% 마진 추가
```

### 낙상 감지 (가속도계 — 실시간, 학습 기반 적응형) — 2026-03-28 재설계

#### 기존 문제
```
기존: 자유낙하(< 5.0g) → 충격(> 15.0g) — 핸드폰 떨어뜨림용
문제 1: 워치는 손목에 붙어있어서 완전한 자유낙하(0g)가 발생 안 함
문제 2: Watch 4 vs Watch 5 센서 감도 차이 → 같은 임계값이 다르게 작동
문제 3: P2P 수신 진동이 가속도계를 흔들어서 수신자에서 낙상 오감지
```

#### 새 설계: 학습 기반 적응형 임계값
```
10분 캘리브레이션에서 가속도계 베이스라인도 함께 학습:
  - accelMean: 평상시 가속도 magnitude 평균 (보통 9.5~10.5)
  - accelStd: 평상시 가속도 표준편차 (기기마다 다름)

적응형 임계값 자동 계산:
  - 떨어짐 임계값 = accelMean - accelStd × 1.5 (범위: 3.0~8.0)
  - 충격 임계값 = accelMean + accelStd × 4 (범위: 12.0~30.0)
    ※ 최소 12.0 (센서 민감도 낮은 기기 대응)
    ※ 이전 15.0 → Watch 5(std=1.0)에서 threshold=15.0 강제되어 감지 실패

예시 (기기별 자동 보정):
  Watch 4: mean=9.8, std=2.0 → fall<6.8, impact>17.8
  Watch 5: mean=9.9, std=1.0 → fall<8.4→8.0, impact>13.9
  → 각 워치 센서 특성에 맞게 자동 보정됨
```

#### 감지 흐름
```
1단계: 떨어짐 감지 — magnitude < fallThreshold (학습 기반)
  조건: HR > 0 (착용 확인)
  조건: 이전 magnitude가 정상 범위 (진동 노이즈 필터)
  조건: P2P 진동 수신 중이 아닐 것 (isReceivingAlert == false)
  조건: 쿨다운/안정화 중이 아닐 것
  조건: WATCH_REMOVED 상태가 아닐 것

2단계: 충격 감지 — magnitude > impactThreshold (학습 기반)
  조건: 떨어짐 후 1초 이내 (FALL_WINDOW_MS = 1000)
    ※ 실제 낙상은 0.3~0.5초 안에 dip→impact 발생
    ※ 1초 넘으면 낙상 아닌 다른 움직임 → 리셋
  조건: 떨어짐 후 최소 50ms 경과 (진동 스파이크 필터)

결과: FALL_DETECTED → "넘어지셨나요?" 즉시 표시 → 5초 확인 대기
  "괜찮아요" → 학습 + 해제
  무응답 → EMERGENCY → P2P BLE 전파
```

#### 진동 오감지 방지 (3중 필터)
```
① BleAlertService.isReceivingAlert → 진동 수신 중이면 낙상 감지 완전 스킵
② lastAccelMagnitude > fallThreshold → 이전값이 정상일 때만 떨어짐 인정
③ 충격 전 최소 50ms 경과 → 진동의 빠른 진동(~ms) 패턴 차단
```

#### 학습 데이터 저장 (SharedPreferences)
```
CalibrationActivity (10분 학습) 저장:
  accelMean: 평상시 가속도 평균
  accelStd: 평상시 가속도 표준편차

SensorService (시작 시) 로드:
  fallThreshold = (accelMean - accelStd * 1.5).coerceIn(3.0, 8.0)
  impactThreshold = (accelMean + accelStd * 4).coerceIn(15.0, 30.0)

학습 전 기본값: mean=9.81, std=2.0 → fall<6.8, impact>17.8
```

### 잠 vs 응급 구분
```
30초 이상 무움직임 + 바이탈 정상 → 수면 의심 (SLEEP_SUSPECTED)
30초 이상 무움직임 + 바이탈 이상 → 응급 (EMERGENCY)
수면 의심 시: 부드러운 반복 알림 (강한 경보 아님)
```

## 이벤트 발생 시 흐름 (시간순)

### 1. 이상 감지 (0초)
```
evaluateState() → NORMAL → MILD_ANOMALY
→ 즉시 WAITING_ACK으로 전환
→ notifyWorker() 호출:
  - 진동 1회 (500ms)
  - fullScreenIntent → AckAlertActivity 실행 (화면 자동 켜짐)
  - 비프 3회
  - 모니터링 1초 간격으로 전환
```

### 2. AckAlertActivity (발신자 화면)
```
┌─────────────────┐
│  괜찮으세요?     │
│                 │
│ [✅ 괜찮아요]   │  학습 O + 경보 해제
│ [⏹ 오작동 종료] │  학습 X + 경보 해제
│ [🚨 즉시 호출]  │  즉시 P2P+서버 경보
│                 │
│ 무응답 시 자동경보│
└─────────────────┘
```

### 3-A. "괜찮아요" 누른 경우
```
→ state = ACKNOWLEDGED
→ ackCooldownUntil = now + 30초
→ 현재 HR/SpO₂/체온 5x 가중 학습 (이 값은 나에게 정상)
→ emergencyVibrator.cancel() (진동 즉시 정지)
→ stopEmergencyBeep() (비프 즉시 정지)
→ BLE cancelEmergency() → "N" 광고 (수신자 자동 정지)
→ 30초간 재에스컬레이션 완전 차단
→ 30초 후 HR 정상이면 NORMAL, 절대상한이면 재알림
```

### 3-B. "오작동 종료" 누른 경우
```
→ state = NORMAL (바로 정상)
→ ackCooldownUntil = now + 30초
→ 학습 안 함 (경보 범위 변화 없음)
→ 진동/비프 즉시 정지
→ BLE cancelEmergency()
```

### 3-C. "즉시 호출" 누른 경우
```
→ state = EMERGENCY
→ BLE broadcastEmergency() → "SPW-001E" 최대출력
→ emergencyVibrate() → 최대 볼륨 연속 비프+진동 (위치 찾기용)
→ 폰+서버에 즉시 전송
```

### 3-D. 무응답 (테스트 5초, 실전 300초)
```
→ isSleeping() 체크
  - 바이탈 정상 + 무움직임 30초 → SLEEP_SUSPECTED (부드러운 알림)
  - 그 외 → EMERGENCY
```

## EMERGENCY 상태 (사고 확정)

### 발신자(사고자) 워치
```
최초 1회 (isEmergencyAlarmActive 플래그):
  1. AckAlertActivity 표시 (fullScreenIntent, CLEAR_TOP)
  2. BLE "SPW-001E" 최대출력 광고 시작
  3. 최대 볼륨 연속 비프 (500ms 비프 + 300ms 무음 반복, 무한)
  4. 최대 강도 연속 진동 (무한 반복)
  ※ "괜찮아요"/"오작동 종료" 누를 때까지 절대 안 멈춤
  ※ 소리로 사고자 위치를 찾을 수 있게 최대 볼륨

매 1초 (상태 업데이트):
  1. 폰에 Wearable 전송 (실시간 바이탈)
  2. 서버에 HTTP 전송 (status="danger")
  3. 대시보드 자동 반영
```

## P2P BLE 수신 (주변 동료)

### 수신자 워치 (Watch 5)
```
BLE 스캔에서 "SPW-001E" 수신:
  1. RSSI → 거리 계산 (경로손실모델: TX_POWER=-65, N=2.5)
  2. 5구간 zone 분류 (ZONE1~ZONE5)
  3. 거리 브로드캐스트 전송 → P2pAlertActivity 실시간 갱신
  4. zone 변경 시 → 진동 패턴 즉시 교체
  5. 첫 1회: fullScreenIntent → P2pAlertActivity 실행 (화면 자동 켜짐)
```

### 5구간 거리별 진동+비프 (수신자)
| 구간 | RSSI | 거리 | 진동 간격 | 비프 간격 | 색상 |
|------|------|------|----------|----------|------|
| ZONE1 | >-50 | ~1m | 100ms on, 50ms off | 200ms | 빨강 |
| ZONE2 | -50~-60 | ~3m | 200ms on, 300ms off | 500ms | 주황 |
| ZONE3 | -60~-67 | ~5m | 200ms on, 1초 off | 1초 | 노랑 |
| ZONE4 | -67~-75 | ~10m | 200ms on, 2초 off | 2초 | 파랑 |
| ZONE5 | <-75 | 10m+ | 200ms on, 4초 off | 4초 | 회색 |

**핵심: 진동 세기는 모두 MAX. 간격만 다르게 → 체감 확실**
**가까이 갈수록 빠르게 → 소리+진동 방향으로 사고자를 찾아감**
**거리는 매 BLE 수신마다 실시간 갱신 (30초 제한 없음)**

### P2pAlertActivity (수신자 화면)
```
┌─────────────────┐
│  🚨 긴급!       │
│  조영진          │  ← 사고자 이름 (서버에서 자동 로드)
│  ~5m            │  ← 실시간 거리 (매 BLE 수신마다 갱신)
│                 │
│ [🏃 응답함]     │  → 진동+비프 정지 + Activity 종료
│ [도움불가]       │  → 진동+비프 정지 + Activity 종료
└─────────────────┘
```

### 거리 실시간 갱신 구조 — 2026-03-31 재설계
```
BLE 수신 (매 ~0.1초) → 거리 계산 → 매번:

  ① EMA 스무딩 (노이즈 제거)
     smoothedDistance = 이전거리 × 0.7 + 측정값 × 0.3
     → 원시 RSSI 튐(1m→8m→2m) 방지
     → 걸어가면 거리가 부드럽게 줄어듦

  ② 스무딩된 거리로 Zone 판정
     → rawZone이 현재와 다르면 "전환 대기" 시작
     → 1초간 유지되면 진짜 전환 (ZONE_HOLD_MS = 1000)
     → 1초 안에 원래로 돌아오면 무시 (노이즈 튐)

  ③ P2P_DISTANCE 브로드캐스트 → P2pAlertActivity 화면 거리 갱신 (실시간)
  ④ Zone 전환 확정 시 → 진동 패턴 교체 + 비프 간격 변경
  ⑤ Zone 동일 시 → 화면만 업데이트 (진동 패턴 유지)

설계 이유:
  - 거리 숫자: 실시간 갱신 (EMA로 부드럽게, 빠르게)
  - 진동 패턴: 1초 유지 후 전환 (몸으로 느끼는 건 안정적이어야)
  - 소리: 발신자 워치 최대 볼륨 비프 (방향 찾기용, 이미 구현)
```

### 경보 해제 조건
```
① 수신자가 "응답함"/"도움불가" 누름 → suppressedWorkerIds에 추가 (30초간 재알림 방지)
② 발신자가 "괜찮아요"/"오작동 종료" 누름 → BLE "N" 광고 → 수신자 자동 정지
```

## HR=0 처리 (워치 벗음/센서 접촉 불량) — 2026-03-28 재설계

### 설계 원칙
**HR=0 = 생체 데이터 없음 → 이상 감지 판단 불가 → 모든 판단 중지**
유일한 예외: 이미 EMERGENCY/FALL_DETECTED인 경우 (의식 잃고 워치 빠짐 가능성)

### 상태별 HR=0 처리 (연속 3회 이상)
```
┌─────────────────────┬──────────────────────────────────────────┐
│ 현재 상태            │ HR=0 발생 시 행동                         │
├─────────────────────┼──────────────────────────────────────────┤
│ EMERGENCY           │ ★ EMERGENCY 유지 (의식 잃고 워치 빠짐)    │
│                     │   P2P BLE + 서버 전송 계속                 │
├─────────────────────┼──────────────────────────────────────────┤
│ FALL_DETECTED       │ ★ 타이머 계속 진행                        │
│                     │   ACK_TIMEOUT 후 EMERGENCY 전환            │
│                     │   (낙상 후 워치 이탈 가능)                 │
├─────────────────────┼──────────────────────────────────────────┤
│ NORMAL              │ → WATCH_REMOVED (즉시, 조용히)            │
│ ACKNOWLEDGED        │ → WATCH_REMOVED (이미 "괜찮아요" 눌렀음)  │
│ MILD_ANOMALY        │ → WATCH_REMOVED (벗으면서 흔들림이 원인)  │
│ WAITING_ACK         │ → WATCH_REMOVED (벗는 중이라 응답 못함)   │
│ SLEEP_SUSPECTED     │ → WATCH_REMOVED (잠자다 벗은 것)          │
└─────────────────────┴──────────────────────────────────────────┘
```

### WATCH_REMOVED 진입 시 즉시 실행
```
1. 진동/비프 즉시 정지
2. BLE 경보 해제 (cancelEmergency)
3. watchRemovedTime 기록 (자동종료 타이머 시작)
4. 모니터링 간격 → 10초 (HR>0 복귀 감지 최소한만)
5. 베이스라인 학습 중지 (updateBaseline 스킵)
6. 서버 전송 중지 (sendToServer 스킵)
7. 적응형 모니터링 레벨 갱신 중지 (updateMonitorLevel 스킵)
```

### WATCH_REMOVED → 자동 작업 종료 (10분)
```
워치 미착용 10분 경과:
  1. AUTO_END_WORK 브로드캐스트 → MainActivity
  2. MainActivity.endWork() 호출:
     - SensorService 정지
     - AlertService 정지
     - BLE 완전 정지 (동료 위험 수신은 폰이 담당)
  3. 서버에 status="offline" 1회 전송
  4. workMode = IDLE
  5. "워치 미착용 10분 → 작업 자동 종료" 표시
```

### 작업 종료 버튼 (수동)
```
endWork() 실행 시:
  1. SensorService 정지 (stopService)
  2. AlertService 정지 (stopService)
  3. BLE 완전 정지 (BleAlertService.stopAll) — 광고+스캔 모두
  4. workMode = IDLE → 대기 화면 표시

※ 워치 벗음/작업 종료 → 워치 BLE 정지 (책상 위 워치가 진동해봐야 의미 없음)
※ 동료 위험 수신 → 폰 BridgeService가 BLE 스캔으로 담당 (폰은 항상 소지)
※ 다시 착용 + 작업 시작 → BLE 재시작
```

### 워치 재착용 시
```
HR > 0 복귀 감지:

Case A — workMode == IDLE (작업 종료 or 자동 종료 후):
  → "작업 시작" 화면 표시 (자동 시작 아님)
  → "워치 착용 감지 — 작업 시작을 눌러주세요" 배너
  → 사용자가 "작업 시작" 눌러야 센서 서비스 시작

Case B — workMode == WORKING (벗었다 다시 착용):
  → 10초 안정화 구간 (stabilizingUntil)
  → 안정화 중: evaluateState()의 이상 감지 비활성
  → "센서 안정화 중 (10초)..." 배너 표시
  → 10초 후 정상 모니터링 재개

※ 안정화 이유: 재착용 직후 센서값이 불안정하여 오경보 발생
※ 10초면 PPG 센서가 피부 접촉 후 안정적 신호를 잡기에 충분
```

### HR=0 심정지 vs 벗음 구분 한계
```
현실적으로 구분 불가능:
  - PPG 센서는 피부 접촉 없으면 0, 심정지여도 혈류 없으면 0
  - 가속도계로 부분 구분 가능:
    · 벗음: 손목 움직임 → 정지 (워치가 테이블에 놓임)
    · 심정지: 갑작스러운 낙하 or 변화 없음

현재 설계 판단:
  - NORMAL → HR=0 = 벗음 (99% 확률, 심정지는 보통 HR 급락이 선행)
  - HR 급락(120→60→0)은 ⑦ 급락 감지(20bpm 이상)에서 먼저 MILD_ANOMALY 트리거
  - 즉, 진짜 심정지면 0이 되기 전에 이상 감지가 먼저 발동됨
```

```
│ ① BLE P2P    │ 1-100ms  │ ★★★★★   │ 주변 동료 경보  │
│ ② 워치→폰→서버│ 100-500ms│ ★★★★☆   │ 서버 모니터링   │
│ ③ 워치→서버   │ 100-1s   │ ★★★☆☆   │ 서버 직접 전송  │
└──────────────┴──────────┴──────────┴───────────────┘

긴급 시: ①②③ 동시 발동 (하나라도 성공하면 됨)
평상시: ② 주로 사용 (폰 중계), ③ 보조 (5분에 1회)
```

### 폰 중계 상세 (WearableListenerService)
```
수신 경로:
  /safepulse/status → handleStatus():
    ① UI 브로드캐스트 (폰 앱 화면 갱신)
    ② 서버 POST /api/workers/:id/sensor (중계)  ← ★ 핵심 추가

  /safepulse/alert → handleAlert():
    ① 폰 진동
    ② 서버 POST /api/alerts/emergency (중계)
    ③ UI 브로드캐스트

  /safepulse/sensor → handleSensorData():
    ① 서버 POST /api/workers/:id/sensor (중계)
    ② UI 브로드캐스트

  /safepulse/ack → handleAck():
    ① 로그만
```

## 2026-03-28 발견 문제 및 개선 이력

### 문제 A: HR=0 오작동 폭주 (해결 완료)
```
증상: 워치 벗을 때 → HR 불안정 → MILD_ANOMALY → HR=0 → WAITING_ACK → EMERGENCY
원인: MILD_ANOMALY/WAITING_ACK에서 HR=0을 위험 신호로 처리
수정: EMERGENCY/FALL_DETECTED 외 모든 상태에서 HR=0 → 즉시 WATCH_REMOVED
파일: SensorService.kt handleHeartRateZero()
```

### 문제 B: 작업 종료 시 서비스 미정지 (해결 완료)
```
증상: 작업 종료 눌러도 SensorService 계속 실행 → 워치 벗으면 오작동
원인: endWork()에서 서비스 정지 코드 없음
수정: endWork()에서 SensorService + AlertService + BLE 완전 정지
파일: MainActivity.kt endWork()
```

### 문제 C: 폰 센서 데이터 서버 중계 누락 (해결 완료)
```
증상: 워치 WiFi 없으면 평상시 센서 데이터가 서버에 안 감
원인: handleStatus()가 UI 브로드캐스트만 하고 서버 중계 안 함
        handleSensorData() 코드는 있지만 워치가 /safepulse/sensor로 안 보냄
수정: handleStatus()에서 서버 POST /api/workers/:id/sensor 추가
      워치 state → 서버 status 변환 (EMERGENCY→danger, FALL_DETECTED→caution 등)
파일: WearableListenerService.kt handleStatus()
```

### 문제 D: 재착용 시 오경보 (해결 완료)
```
증상: 워치 다시 착용하면 센서 안정화 전 불안정한 HR로 오경보
원인: WATCH_REMOVED → HR>0 → 즉시 NORMAL + evaluateState 재개
수정: 10초 안정화 구간 (stabilizingUntil) 동안 이상 감지 비활성
파일: SensorService.kt evaluateState() WATCH_REMOVED/NORMAL
```

### 문제 E: FALL_DETECTED에서 HR=0 시 타이머 미진행 (해결 완료)
```
증상: 낙상 후 HR=0이면 evaluateState 가드에 걸려 타이머 안 돌아감
원인: evaluateState() 시작의 if (heartRate <= 0) return 가드
수정: FALL_DETECTED/EMERGENCY는 HR=0이어도 evaluateState 진입 허용
파일: SensorService.kt evaluateState() hrZeroButTimerNeeded 플래그
```

### 문제 F: Watch 4/5 센서 감도 차이로 낙상 감지 불일치 (해결 완료)
```
증상: Watch 4는 세게 떨어뜨려도 안 잡히고, Watch 5는 진동만으로 잡힘
원인: 기기별 가속도계 하드웨어 감도 차이 + 하드코딩 임계값
수정: 10분 캘리브레이션에서 가속도 베이스라인(mean/std) 학습
      → fallThreshold = mean - std*1.5, impactThreshold = mean + std*4
      → 기기별 자동 보정
파일: CalibrationActivity.kt (학습), SensorService.kt (적용)
```

### 문제 G: 수신자 진동이 가속도계 오염 → 자체 낙상 (해결 완료)
```
증상: P2P 수신 진동 중 수신자 워치가 "넘어졌나요?" 표시
원인: 진동 모터가 가속도계를 흔들어 떨어짐+충격 패턴 생성
수정: BleAlertService.isReceivingAlert 플래그
      → 진동 수신 중이면 SensorService 낙상 감지 완전 스킵
파일: BleAlertService.kt (플래그), SensorService.kt (체크)
```

### 문제 H: 긴급 화면 안 열림 — Wear OS BAL 제한 (해결 완료)
```
증상: 화면 꺼진 상태에서 EMERGENCY/P2P → 화면 안 뜸
원인: Wear OS Background Activity Launch 제한
수정: Activity 시작 전 WakeLock으로 화면 먼저 켜기
      → wakeAndStartActivity() 유틸 함수
파일: SensorService.kt, BleAlertService.kt
```

### 문제 I: 수신자 P2pAlertActivity 발신자 해제 시 안 닫힘 (해결 완료)
```
증상: 발신자 "오작동 종료" → 수신자 진동 멈추지만 화면 그대로
원인: BLE "N" 수신 시 진동만 취소, Activity 닫기 코드 없음
수정: CLOSE_P2P 브로드캐스트 → P2pAlertActivity closeReceiver → finish()
파일: BleAlertService.kt, P2pAlertActivity.kt
```

### 문제 J: AlertService 서버 danger alert 반복 진동 (해결 완료)
```
증상: 오작동 종료 눌러도 서버에 남은 danger alert를 10초마다 가져와서 반복
원인: AlertService가 서버 danger alert를 무조건 진동+표시
수정: danger 레벨 전부 스킵 (긴급은 BLE P2P + SensorService가 처리)
파일: AlertService.kt
```

### 문제 K: BLE 스캔 재시작 안 됨 — stopAll 후 스캔 불능 (해결 완료)
```
증상: endWork() → stopAll() 후 다시 작업 시작 → BLE 스캔 수신 0건
      앱을 강제로 열어야(MainActivity) 수신 시작됨
원인: stopScanning()이 scanCallbackInstance를 null로 안 함
      → startScanning() 재호출 시 이전 callback 잔재로 시스템이 새 스캔 무시
수정: startScanning()에서 기존 스캔 확실히 정리(stopScan + callback=null) 후 재시작
      stopScanning()에서도 callback=null 처리
파일: BleAlertService.kt startScanning(), stopScanning()
```

### 문제 L: P2P 수신자 워치 화면 안 뜸 (해결 완료)
```
증상: 수신자가 BLE P2P 수신 + notification posted → fullScreenIntent로 화면 안 켜짐
원인: BleAlertService는 object(싱글톤)이지 Android Foreground Service가 아님
수정: BleAlertService → SensorService(ACTION_LAUNCH_P2P)로 위임
      SensorService(Foreground Service)에서 fullScreenIntent notification 발행
파일: BleAlertService.kt, SensorService.kt
```

### 문제 M: 폰 앱 백그라운드 미작동 + 지도 자동 실행 (해결 완료)
```
증상 1: 앱 닫으면 BLE P2P 수신 안 됨
증상 2: P2P 수신 시 지도가 즉시 자동 실행 (확정 전인데 놀람)
원인 1: BridgeService가 MainActivity에서만 시작됨 + BootReceiver 없음
원인 2: BridgeService가 EmergencyMapActivity를 직접 startActivity
수정:
  - BootReceiver 추가 → 폰 재부팅 시 BridgeService 자동 시작
  - EmergencyMapActivity 자동 실행 제거
  - P2P notification (IMPORTANCE_HIGH + fullScreenIntent)으로 변경
  - "위치 보기" 누를 때만 지도 표시
  - 발신자 해제(BLE "N") → notification 자동 제거 + 진동 정지
  - 발신자 폰 긴급 중 뒤로가기 차단
파일: BootReceiver.kt(신규), AndroidManifest.xml, BridgeService.kt, MainActivity.kt
```
