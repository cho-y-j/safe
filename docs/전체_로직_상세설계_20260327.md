# SafePulse 전체 로직 상세 설계

## 시스템 구성
```
[Watch 4 W-001 조영진] ←BT→ [S22 폰]
[Watch 5 W-002 이서준] ←BT→ [Note9 폰]
[서버 192.168.0.9:4000] ←WiFi→ 모든 기기
[대시보드 웹] ←Socket.io→ 서버
```

---

## 1단계: 평상시 모니터링

### 워치 (SensorService)
```
매 30초 (ACTIVE 레벨):
  1. 심박 센서 읽기 → heartRate
  2. SpO₂ 센서 읽기 → spo2
  3. 가속도계 → activityLevel (안정/활동 구분)
  4. updateBaseline() → 개인 베이스라인 EMA 업데이트
  5. evaluateState() → 현재 상태 판단
  6. broadcastStatus() → MainActivity UI 업데이트 + Wearable로 폰에 전송
  7. sendToServer() → 서버에 센서 데이터 POST (status 포함)
```

### BLE 광고 (평상시)
```
BleAlertService.startNormalAdvertise():
  - 제조사 데이터: "SPW-001N" (0xDADA)
  - 모드: LOW_POWER
  - 출력: LOW
```

### BLE 스캔 (항상)
```
BleAlertService.startScanning():
  - 필터: 0xDADA
  - 모드: LOW_LATENCY
  - 수신 시: processScanResult() → "N"이면 무시, "E"면 긴급 처리
```

## 2단계: 이상 감지 (evaluateState)

### NORMAL → MILD_ANOMALY 조건
```
다음 중 하나라도 해당:
  - HR > 안정평균 + alertRangeUpper (직업별: 40/55/65/80)
  - HR < 안정평균 - alertRangeLower (30)
  - HR >= 180 (절대 상한)
  - HR < 40 (절대 하한)
  - SpO₂ < 개인평균 - alertSpo2Range (직업별: 3/4/5/6%)
  - 체온 > 개인평균 + alertTempRange (직업별: 0.5/0.8/1.0/1.5°C)
  - 심박 급락 (이전 대비 20bpm 이상)
```

### MILD_ANOMALY → WAITING_ACK (즉시)
```
notifyWorker() 호출:
  - 진동 1회 (500ms)
  - fullScreenIntent → AckAlertActivity 실행
  - 비프 3회
```

## 3단계: AckAlertActivity (발신자 화면)

### 화면
```
┌─────────────────┐
│  괜찮으세요?     │
│                 │
│ [✅ 괜찮아요]   │  → ACTION_ACKNOWLEDGE
│ [⏹ 오작동 종료] │  → ACTION_DISMISS
│ [🚨 즉시 호출]  │  → ACTION_MANUAL_EMERGENCY
│                 │
│ 무응답 시 자동경보│
└─────────────────┘
```

### "괜찮아요" (ACTION_ACKNOWLEDGE)
```
onAcknowledge():
  1. currentState = ACKNOWLEDGED
  2. ackCooldownUntil = now + 30초
  3. isEmergencyAlarmActive = false
  4. 현재 HR/SpO₂/체온 → 5x 가중 학습 (이 값은 나에게 정상)
  5. stopEmergencyBeep()
  6. BleAlertService.cancelEmergency() → BLE "N" 광고로 전환
```

### "오작동 종료" (ACTION_DISMISS)
```
onDismiss():
  1. currentState = NORMAL (ACKNOWLEDGED가 아닌 NORMAL!)
  2. ackCooldownUntil = now + 30초
  3. isEmergencyAlarmActive = false
  4. 학습 안 함 (센서 오류이므로)
  5. stopEmergencyBeep()
  6. BleAlertService.cancelEmergency()
```

### "즉시 호출" (ACTION_MANUAL_EMERGENCY)
```
onManualEmergency():
  1. currentState = EMERGENCY
  2. ackCooldownUntil = 0 (쿨다운 리셋)
  3. BleAlertService.broadcastEmergency() → BLE "E" 광고
  4. emergencyVibrate() → 최대 볼륨 연속 비프+진동
```

### 무응답 (5초 — 테스트용, 실전 300초)
```
WAITING_ACK → 5초 경과:
  - isSleeping() 체크
    - YES → SLEEP_SUSPECTED
    - NO → EMERGENCY
```

## 4단계: EMERGENCY (발신자)

### 최초 1회만 (isEmergencyAlarmActive == false)
```
1. isEmergencyAlarmActive = true
2. fullScreenIntent → AckAlertActivity (CLEAR_TOP)
3. BleAlertService.broadcastEmergency() → BLE "SPW-001E" 최대출력
4. emergencyVibrate() → 무한반복 진동 + 연속 비프 (위치 찾기용)
```

### 매 루프 (1초마다)
```
1. WearableCommService.sendEmergency() → 폰에 전송
2. ServerClient.sendEmergencyAlert() → 서버에 전송
3. sendToServer(status="danger") → 대시보드 반영
```

### ACK 쿨다운 체크 (EMERGENCY case 최상단)
```
if (now < ackCooldownUntil):
  → currentState = ACKNOWLEDGED
  → cancelEmergency()
  → return (EMERGENCY 로직 스킵)
```

## 5단계: ACKNOWLEDGED (ACK 후)

### ★★★ 현재 버그: 재에스컬레이션 ★★★
```
현재:
  ACKNOWLEDGED case에서:
    - HR >= 180 || HR < 40 || SpO₂ <= 90 → MILD_ANOMALY (재알림)
    - 정상 범위 복귀 → NORMAL

문제:
  HR이 여전히 alertRange 밖이면 → 정상 복귀 조건 불충족
  → ACKNOWLEDGED 유지되지만, 30초 쿨다운 끝나면?
  → 다시 evaluateState에서 ACKNOWLEDGED case
  → HR 여전히 높으면 → 절대상한 아니므로 유지
  → 하지만 30초 후 MILD_ANOMALY → WAITING_ACK → EMERGENCY 무한루프

해결:
  ackCooldownUntil 동안은 ACKNOWLEDGED에서 MILD_ANOMALY 전환도 차단
  OR
  "괜찮아요" 학습으로 alertRange가 확장되었으므로 같은 HR에서 재트리거 안 됨
  (하지만 학습 EMA가 느려서 즉시 반영 안 될 수 있음)

최종 해결:
  ACKNOWLEDGED case에서 ackCooldownUntil 체크 추가:
  if (now < ackCooldownUntil) return  // 쿨다운 중이면 아무것도 안 함
```

## 6단계: 수신자 워치 (Watch 5)

### BLE "E" 수신 (processScanResult)
```
1. 제조사 데이터 "SPW-001E" 파싱 → workerId=W-001, status=E
2. suppressedWorkerIds 체크 → 이미 dismiss했으면 무시
3. RSSI → 거리 계산 (TX_POWER=-65, N=2.5)
4. 5구간 zone 분류 (ZONE1~ZONE5)
5. 거리 브로드캐스트 전송 (P2P_DISTANCE) → P2pAlertActivity 실시간 갱신
6. zone 변경 시: 진동 패턴 즉시 교체 + 비프 간격 변경
7. 첫 1회만: fullScreenIntent → P2pAlertActivity 실행
```

### ★★★ 수신자에게 AckAlertActivity가 뜨면 안 됨 ★★★
```
현재 버그:
  Watch 5의 SensorService도 독립적으로 evaluateState() 실행
  → Watch 5 착용자의 HR이 높으면 (흥분/운동) → MILD_ANOMALY → AckAlertActivity
  → P2pAlertActivity와 혼재

이건 정상 동작임 (Watch 5 착용자 본인의 이상 감지)
하지만 P2P 수신 중에 AckAlertActivity가 뜨면 혼란

해결:
  P2P 수신 중 (activeEmergencyWorkers가 비어있지 않으면)
  → 본인의 MILD_ANOMALY 판정을 일시 보류
  OR
  AckAlertActivity가 P2pAlertActivity 위에 뜨지 않게 분리
  (이미 다른 태스크이므로 동시에 안 보임 — 이건 Android가 처리)
```

## 7단계: P2pAlertActivity (수신자 화면)

### 화면
```
┌─────────────────┐
│  🚨 긴급!       │
│  조영진          │  ← 사고자 이름
│  ~5m            │  ← 실시간 거리 (브로드캐스트 수신)
│                 │
│ [🏃 응답함]     │  → dismiss + 진동/비프 정지
│ [도움불가]       │  → dismiss + 진동/비프 정지
└─────────────────┘
```

### 거리 실시간 갱신
```
P2P_DISTANCE 브로드캐스트 수신 → updateDistance()
  - 거리 텍스트 갱신
  - 색상 갱신 (빨/주/노/파/회)
```

### "응답함" / "도움불가"
```
BleAlertService.dismissReceivedAlertFull(workerId):
  1. suppressedWorkerIds에 추가 (30초간 재알림 방지)
  2. activeVibrator.cancel() → 진동 즉시 정지
  3. stopBeep() → 비프 정지
  4. currentZone = "" / alertActivityLaunched = false (리셋)
```

### BLE "N" 수신 (발신자가 해제한 경우)
```
processScanResult에서 status == 'N':
  → activeVibrator.cancel()
  → activeEmergencyWorkers.remove()
  → stopBeep()
  → 자동 정지 (사용자 조작 불필요)
```

## 9단계: 폰 (수신자 — Note9)

### BLE 수신 (BridgeService)
```
BLE scanCallback에서 "SPW-001E" 수신:
  1. isMyWatch 체크 (myWorkerId와 비교)
     - myWorkerId == workerId → 발신자 폰 → 스킵
     - myWorkerId != workerId → 수신자 폰 → P2P 처리
  2. 진동 + 비프 (10초 중복 방지)
  3. MainActivity 실행 (p2p_emergency intent)
  4. P2P_EMERGENCY 브로드캐스트 → p2pCard 표시
  5. launchEmergencyMap(workerId) → 서버에서 위치 조회 → 지도
```

### P2P 카드 (수신자 폰)

## 5구간 거리별 진동 (수신자)

| 구간 | RSSI | 거리 | 진동 간격 | 비프 간격 | 색상 |
|------|------|------|----------|----------|------|
| ZONE1 | >-50 | ~1m | 100ms on, 50ms off | 200ms | 빨강 |
| ZONE2 | -50~-60 | ~3m | 200ms on, 300ms off | 500ms | 주황 |
| ZONE3 | -60~-67 | ~5m | 200ms on, 1초 off | 1초 | 노랑 |
| ZONE4 | -67~-75 | ~10m | 200ms on, 2초 off | 2초 | 파랑 |
| ZONE5 | <-75 | 10m+ | 200ms on, 4초 off | 4초 | 회색 |

**발신자(사고자)는 구간 무관 — 항상 최대 볼륨 연속 비프+진동**

## 개인화 학습

### "괜찮아요" 누를 때
```
현재 HR → restHrHistory 또는 activeHrHistory에 5x 가중 추가
현재 SpO₂ → spo2History에 5x 가중 추가
현재 체온 → tempHistory에 5x 가중 추가
→ EMA 업데이트 → 경보 범위 확장 → 같은 값에서 재경보 안 남
```

### "오작동 종료" 누를 때
```
학습 안 함 → 경보 범위 변화 없음 → 같은 값에서 다시 경보 남
```

### 저장/복원
```
SharedPreferences에 저장:
  baselineHR, restHrStd, activeHrMean, activeHrStd
  spo2Mean, spo2Std, baselineTempMean, baselineTempStd
앱 재시작 시 복원
```

## 현재 남은 버그

### 버그 1: ACKNOWLEDGED에서 재에스컬레이션
**위치:** SensorService.kt evaluateState() ACKNOWLEDGED case
**해결:** `if (now < ackCooldownUntil) return` 추가

### 버그 2: 발신자 진동이 ACK 후에도 안 멈춤
**위치:** emergencyVibrate()의 vibrator.vibrate(repeat=0)이 cancel 안 됨
**해결:** vibrator 참조 저장 → onAcknowledge/onDismiss에서 vibrator.cancel() 명시 호출

### 버그 3: 수신자 워치에 AckAlertActivity 뜸
**원인:** Watch 5 착용자 본인의 HR이 높으면 본인 이상 감지 → AckAlertActivity
**해결:** 이건 정상 동작이지만, P2P 수신 중에는 본인 이상 감지 일시 보류 (또는 구분 UI)
